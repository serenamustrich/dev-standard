## 一、概述

### 1.1 背景

本规范定义了AI多Agent团队如何协作开发软件，并如何通过**需求管理系统**进行全流程管理。

**需求管理系统**是运行在本地Docker上的Web系统，提供需求管理、Bug管理、版本管理、工时统计等功能。

### 1.2 目标

- 建立清晰的AI开发组织结构和职责分工
- 定义标准化的开发工作流程
- 通过多Agent协作减少AI幻觉，提高开发质量
- 实现软件开发全流程的可追溯和可管理
- 所有工作记录在需求管理系统中留痕

### 1.3 适用范围

适用于所有使用AI Agent进行软件开发的项目。

### 1.4 规范位置

**开发规范（文档）**：`code/开发规范/`（全局）
**需求管理系统（系统）**：本地Docker服务

## 二、需求管理系统

### 2.1 系统简介

需求管理系统是团队协作的**核心工具**，所有AI Agent的工作输出都需要在该系统中留痕。

**功能模块**：
- 需求管理 - 创建、评审、追踪需求
- Bug管理 - 提交、追踪、修复Bug
- 版本管理 - 版本规划、发布追踪
- 工时统计 - 任务工时、效率分析
- 评论系统 - 沟通留痕

### 2.2 API接口

**基础URL**：`http://localhost:8080/api`

**认证**：
```
POST /api/auth/login
Body: { "username": "xxx", "password": "xxx" }
Response: { "token": "xxx" }
```

**需求管理**：
```
GET    /api/requirements          # 获取需求列表
GET    /api/requirements/{id}      # 获取需求详情
POST   /api/requirements           # 创建需求
PUT    /api/requirements/{id}       # 更新需求
PUT    /api/requirements/{id}/status  # 变更状态
```

**Bug管理**：
```
GET    /api/bugs                   # 获取Bug列表
POST   /api/bugs                   # 创建Bug
PUT    /api/bugs/{id}             # 更新Bug
```

**版本管理**：
```
GET    /api/versions               # 获取版本列表
POST   /api/versions               # 创建版本
PUT    /api/versions/{id}          # 更新版本
```

**评论**：
```
GET    /api/requirements/{id}/comments   # 获取评论
POST   /api/requirements/{id}/comments   # 添加评论
```

**仪表盘**：
```
GET    /api/dashboard/stats         # 获取统计数据
```

### 2.3 使用规则

**强制留痕**：
- 每个需求创建、更新、状态变更都必须记录
- 每个Bug的创建、修复、关闭都必须记录
- 每次评审结论必须记录
- 所有沟通结论必须通过评论记录

## 三、AI开发组织结构

### 3.1 组织架构图

```
                    ┌─────────────────────────────────────┐
                    │           主Agent (总指挥)            │
                    │  - 任务分解与分配                    │
                    │  - 进度监控与协调                    │
                    │  - 质量把控与决策                    │
                    └─────────────────┬───────────────────┘
                                      │
         ┌────────────────────────────┼────────────────────────────┐
         │                            │                            │
         ▼                            ▼                            ▼
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   需求Agent    │      │   UI设计Agent   │      │  开发Agent     │
│ - 需求分析      │◄────►│ - 界面设计      │◄────►│ - 架构设计     │
│ - 需求建模      │      │ - 交互设计      │      │ - 代码实现     │
│ - 需求录入系统  │      │ - 设计录入系统  │      │ - 代码录入系统  │
└─────────────────┘      └─────────────────┘      └─────────────────┘
```

### 3.2 Agent职责与系统操作

#### 主Agent（总指挥）
- 接收用户需求
- 分解任务分配给SubAgent
- 监控进度和质量
- **系统操作**：查看仪表盘、审核需求

#### 需求Agent
- 进行需求分析和建模
- **系统操作**：创建需求、更新需求、添加评论

#### UI设计Agent
- 设计界面方案
- **系统操作**：在需求下添加设计说明评论

#### 开发Agent
- 进行技术设计
- 编写代码
- **系统操作**：更新需求状态、创建Bug、添加评论

#### 质量Agent
- 设计测试用例
- 执行测试
- **系统操作**：创建Bug、更新Bug状态

#### 部署Agent
- 执行部署
- **系统操作**：更新版本状态、记录部署结果

#### 知识管理Agent
- 整理文档
- **系统操作**：在系统中记录总结

## 四、开发工作流程

### 4.1 标准流程

```
需求分析 → 需求评审 → UI设计 → UI评审 → 开发设计 → 开发评审 → 开发自测 → 测试 → 部署 → 知识沉淀
```

### 4.2 各阶段与系统操作

#### 阶段1：需求分析
**Agent**：需求Agent
**系统操作**：创建需求记录

#### 阶段2：需求评审
**Agent**：主Agent + 需求Agent
**系统操作**：更新需求状态、添加评审评论

#### 阶段3：UI设计
**Agent**：UI设计Agent
**系统操作**：在需求下添加设计说明

#### 阶段4：UI评审
**Agent**：主Agent + UI设计Agent
**系统操作**：添加评审结论评论

#### 阶段5：开发设计
**Agent**：开发Agent
**系统操作**：在需求下添加技术设计说明

#### 阶段6：开发评审
**Agent**：主Agent + 开发Agent
**系统操作**：添加评审结论评论

#### 阶段7：开发自测
**Agent**：开发Agent
**系统操作**：创建Bug、更新需求状态

#### 阶段8：测试
**Agent**：质量Agent
**系统操作**：创建Bug、验证Bug修复

#### 阶段9：部署
**Agent**：部署Agent
**系统操作**：更新版本状态

#### 阶段10：知识沉淀
**Agent**：知识管理Agent
**系统操作**：在系统中记录项目总结

## 五、强制执行规则

### 5.1 每次对话必须调用工作流

```
用户输入 → 调用工作流技能 → 分析请求类型 → 确定流程阶段 → 在需求管理系统中留痕
```

### 5.2 必须留痕的场景

| 场景 | 系统操作 |
|------|----------|
| 接受新需求 | 创建需求记录 |
| 开始开发 | 更新需求状态+添加评论 |
| 发现问题 | 创建Bug记录 |
| 完成功能 | 更新需求状态+添加评论 |
| 测试发现Bug | 创建Bug记录 |
| Bug修复 | 更新Bug状态+添加评论 |
| 部署完成 | 更新版本状态 |

### 5.3 禁止行为

- ❌ 直接开发不创建需求记录
- ❌ Bug修复不更新系统记录
- ❌ 跳过系统留痕直接提交代码

## 六、避免AI幻觉策略

### 6.1 多Agent交叉验证

关键决策由多个Agent独立分析后验证。

### 6.2 事实核查机制

- 所有技术方案必须基于现有代码库
- 所有API调用必须验证可用性
- 所有数据必须标注来源

### 6.3 人机协作确认

关键节点必须人工确认：
1. 需求评审通过
2. 测试通过
3. 部署上线
