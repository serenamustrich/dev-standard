## 3. 需求评审流程

### 3.1 评审类型

| 评审类型 | 评审人 | 评审内容 |
|---------|--------|---------|
| 技术评审 | 技术负责人 | 技术可行性、架构影响 |
| 产品评审 | 产品经理 | 需求完整性、优先级 |
| 设计评审 | UI/UX设计师 | 交互体验、视觉设计 |
| 测试评审 | 测试负责人 | 测试可行性 |

### 3.2 评审流程

```
1. 需求提出
   └── 产品经理创建需求文档

2. 技术预审
   └── 技术负责人评估技术可行性
   └── 输出：技术评估意见

3. 产品评审
   └── 评审需求完整性
   └── 确认优先级
   └── 输出：评审结论（通过/需修改）

4. 设计评审（如需要）
   └── UI/UX设计师提供设计方案

5. 测试评审
   └── 测试负责人评估测试范围

6. 评审通过
   └── 需求进入开发阶段
```

### 3.3 评审通过标准

- 技术可行性确认
- 需求描述清晰完整
- 验收标准明确
- 优先级已确认
- 测试范围已评估

## 4. 需求状态管理

### 4.1 状态定义

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| 待评审 | 需求已创建，等待评审 | 创建需求后 |
| 评审中 | 正在进行评审 | 开始评审流程 |
| 评审通过 | 评审已完成，等待开发 | 评审通过 |
| 开发中 | 正在开发 | 开始开发 |
| 测试中 | 正在测试 | 开发完成 |
| 已完成 | 功能已上线 | 测试通过并发布 |
| 已关闭 | 需求取消 | 明确不做的需求 |

### 4.2 状态流转

```
待评审 → 评审中 → 评审通过 → 开发中 → 测试中 → 已完成
   ↓          ↓
评审不通过   已关闭
     ↓
  待评审（修改后重新提交）
```

## 5. 需求变更管理

### 5.1 变更触发条件

- 用户提出新需求
- 技术方案调整
- 业务需求调整
- 紧急修复

### 5.2 变更流程

```
1. 变更提出
   └── 填写变更申请

2. 影响评估
   └── 评估对进度、成本的影响

3. 变更审批
   └── P0/P1变更：需项目负责人审批
   └── P2/P3变更：可由技术负责人审批

4. 变更执行
   └── 更新需求文档
   └── 通知相关方

5. 变更记录
   └── 在变更记录表中登记
```

### 5.3 紧急变更

对于紧急变更：
- 可先执行后补流程
- 24小时内补全变更文档
- 通知所有相关方

## 6. 需求追踪

### 6.1 追踪维度

- **需求 → 开发任务**：每个功能点对应开发任务
- **需求 → 测试用例**：每个功能点对应测试用例
- **需求 → 代码提交**：代码提交关联需求编号

### 6.2 追踪方式

在代码提交时，关联需求编号：
```bash
git commit -m "feat(wms-manual): 添加流程图功能

关联需求: PRD-2026-002"
```

## 7. 需求优先级定义

| 优先级 | 定义 | 说明 |
|-------|------|------|
| P0 | 紧急且重要 | 阻塞性问题，必须立即处理 |
| P1 | 重要 | 核心功能，必须在当前版本实现 |
| P2 | 中等 | 重要功能，可安排到下一版本 |
| P3 | 低 | 优化功能，可延后处理 |

## 8. 常见问题

### Q: 如何处理需求变更？

A: 按照变更流程执行，评估影响并获得审批后更新需求文档。

### Q: 需求评审不通过怎么办？

A: 根据评审意见修改需求文档，重新提交评审。

### Q: 多个需求冲突怎么办？

A: 由产品经理根据业务价值确定优先级，或由项目负责人决策。
