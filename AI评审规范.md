# AI评审规范

**文档版本**：v1.0
**创建日期**：2026-03-27
**作者**：技术总监

---

## 一、概述

### 1.1 背景

传统的评审流程需要人工参与，但在AI团队协作中，评审环节由所有AI Agent共同完成。每个岗位的AI Agent都需要从自己的职责角度提出意见，只有**全体通过**才能进入下一阶段。

### 1.2 目标

- 建立AI自我评审机制
- 确保每个AI Agent都能从职责角度审查需求
- 实现全员参与的科学评审流程
- 避免AI幻觉，提高需求质量

---

## 二、评审参与角色

### 2.1 评审委员会

每个需求的评审需要以下AI Agent参与：

| 角色 | 评审视角 | 评审重点 |
|------|----------|----------|
| **需求Agent** | 需求完整性 | 需求描述是否清晰、完整、无歧义 |
| **UI设计Agent** | 设计可行性 | 需求中的UI部分是否可实现 |
| **开发Agent** | 技术可行性 | 技术方案是否合理、风险评估 |
| **质量Agent** | 测试可行性 | 是否可测试、测试策略 |
| **部署Agent** | 部署可行性 | 部署复杂度、环境要求 |
| **主Agent** | 综合评估 | 整体可行性、优先级、资源需求 |

### 2.2 评审角色职责

#### 需求Agent评审职责
- 需求描述是否清晰易懂
- 需求范围是否明确
- 验收标准是否可量化
- 需求之间是否有冲突
- 需求优先级是否合理

#### UI设计Agent评审职责
- UI需求是否完整
- 交互流程是否合理
- 设计规范是否一致
- 用户体验是否考虑

#### 开发Agent评审职责
- 技术方案是否可行
- 技术风险是否识别
- 依赖关系是否明确
- 实现复杂度评估
- 代码规范是否符合

#### 质量Agent评审职责
- 功能是否可测试
- 测试用例是否可设计
- 测试策略是否合理
- 质量标准是否明确

#### 部署Agent评审职责
- 部署流程是否清晰
- 环境要求是否明确
- 运维成本评估
- 监控告警是否考虑

#### 主Agent评审职责
- 整体可行性判断
- 资源需求评估
- 优先级建议
- 时间预估
- 风险等级评估

## 三、评审流程

### 3.1 标准评审流程

```
需求创建 → 发布评审通知 → 各Agent独立评审 → 汇总意见 → 全体通过?
                                                                          ↓
                                              ┌────────────────────────────┘
                                              ↓
                                    ┌─────────┴─────────┐
                                    ↓                   ↓
                                  是                  否
                                    ↓                   ↓
                              进入下一阶段        返回需求Agent修改
                                              ↓
                                        重新发布评审
```

### 3.2 评审步骤

#### 步骤1：发布评审通知

需求Agent创建需求后，在需求管理系统中发布评审通知：
```
评审主题：【PRD-XXX】需求名称 - 待评审
评审参与：需求Agent、UI设计Agent、开发Agent、质量Agent、部署Agent、主Agent
截止时间：XXX
```

#### 步骤2：各Agent独立评审

每个Agent从自己的职责角度评审，输出评审意见：

```markdown
## 评审意见

**评审人**：需求Agent
**评审时间**：YYYY-MM-DD HH:mm
**评审结论**：✅ 通过 / ❌ 不通过

### 评审意见
1. [评审点1]
2. [评审点2]

### 建议
- [建议1]
- [建议2]
```

#### 步骤3：汇总意见

主Agent汇总所有评审意见，判断是否通过：

| 情况 | 处理方式 |
|------|----------|
| 全部通过 | 进入下一阶段 |
| 部分不通过 | 返回需求Agent修改 |
| 存在重大问题 | 重新设计 |

#### 步骤4：输出评审报告

```markdown
# 评审报告

**需求编号**：PRD-XXX
**评审时间**：YYYY-MM-DD HH:mm
**评审结论**：✅ 通过 / ❌ 不通过

## 评审汇总

| Agent | 评审结论 | 主要意见 |
|-------|----------|----------|
| 需求Agent | ✅ | ... |
| UI设计Agent | ✅ | ... |
| 开发Agent | ❌ | ... |
| ... | ... | ... |

## 修改要求

[如果不通过，列出需要修改的内容]

## 下一步

[通过后的行动]
```

## 四、评审标准

### 4.1 通过标准

所有AI Agent都输出"通过"，需求才能进入下一阶段。

### 4.2 不通过标准

以下任一情况视为不通过：
- 任何一个Agent评审结论为"不通过"
- 存在重大风险未识别
- 关键信息缺失

### 4.3 评审时效

| 评审类型 | 建议时长 |
|----------|----------|
| 需求评审 | 30分钟内 |
| 设计评审 | 1小时内 |
| 开发评审 | 30分钟内 |
| 测试评审 | 30分钟内 |

## 五、评审规则

### 5.1 强制规则

**⚠️ 所有AI Agent必须参与评审**

**⚠️ 每个Agent必须从职责角度提出意见**

**⚠️ 全体通过才能进入下一阶段**

**⚠️ 不通过必须重新修改需求**

### 5.2 评审质量要求

- 评审意见必须有实质性内容
- 禁止敷衍式评审（"同意"、"通过"等无效评审）
- 每个评审意见必须说明理由
- 不通过必须给出具体修改建议

### 5.3 评审记录

所有评审意见必须记录在需求管理系统的评论中：
- 评审人
- 评审时间
- 评审结论
- 评审意见
- 修改建议

---

## 六、特殊规则

### 6.1 紧急需求

紧急需求可简化评审流程，但仍需各Agent确认：
```
需求Agent快速创建 → 各Agent快速确认 → 主Agent批准 → 进入开发
```

### 6.2 需求变更

需求变更需要重新评审，流程同上。

### 6.3 评审超时

如果某个Agent超时未评审，主Agent可催促或指定替代评审。

## 七、示例

### 示例1：需求评审

**需求**：PRD-2026-005 需求管理系统功能增强

**评审过程**：

#### 需求Agent评审
```
评审结论：✅ 通过

评审意见：
1. 需求描述清晰，包含了4个主要功能模块
2. 验收标准明确，可量化
3. 优先级合理

建议：可以在工时统计模块增加日报生成功能
```

#### UI设计Agent评审
```
评审结论：✅ 通过（附建议）

评审意见：
1. 任务管理模块需要设计任务卡片样式
2. 工时统计模块需要设计统计图表

建议：在详细设计阶段补充UI设计稿
```

#### 开发Agent评审
```
评审结论：❌ 不通过

评审意见：
1. 缺少API返回数据结构定义
2. 分页查询未说明
3. 数据库表结构未设计

建议：需要补充技术设计文档
```

#### 质量Agent评审
```
评审结论：✅ 通过（附建议）

评审意见：
1. 测试用例模块已有，可以复用
2. 需要增加自动化测试接口

建议：在测试模块增加API自动化测试用例
```

#### 部署Agent评审
```
评审结论：✅ 通过

评审意见：
1. 功能不涉及服务器配置变更
2. 数据库迁移需要设计

建议：需要补充数据库迁移脚本
```

#### 主Agent评审
```
评审结论：❌ 不通过

评审意见：
1. 开发Agent提出的技术问题需要解决
2. 需要先完成技术设计再进入开发

建议：
- 返回开发Agent要求补充技术设计
- 重新评审后再进入开发阶段
```

**评审结论**：❌ 不通过

**下一步**：返回需求Agent，由开发Agent补充技术设计文档，重新评审。
